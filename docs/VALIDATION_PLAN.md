# VALIDATION_PLAN.md

自治体生成AIレーダー（Municipal GenAI Radar）の初期検証計画。

目的は、Webサイトやアプリを作る前に、公開情報から50案件を収集し、
「継続追跡できるデータベース事業」として成立しうるデータの密度・継続性があるかを確認することである。

プロジェクトの目的・対象範囲は [PROJECT.md](../PROJECT.md) を、
データの収集・入力ルールは [data/validation/README.md](../data/validation/README.md) を参照すること。

## 1. 検証目標と合格条件

### 検証目標
公開情報から50案件を収集し、data/validation/cases.csv に記録する。

### 対象・対象外
PROJECT.md の「初期対象」「初期対象外」の定義に従う。

### 合格条件
以下をすべて満たした場合、データベース事業として成立する可能性があると判断する。

| # | 条件 | 基準 |
|---|------|------|
| 1 | 自治体と企業を特定できる案件 | 45件以上 |
| 2 | 製品またはサービスを特定できる案件 | 35件以上 |
| 3 | 案件の進捗段階を判定できる案件 | 40件以上 |
| 4 | 複数の公式出典を確認できる案件 | 20件以上 |
| 5 | 同一案件の続報を確認できる案件 | 15件以上 |
| 6 | 費用または定量成果を確認できる案件 | 10件以上 |

各条件の判定方法は「4. 判定基準」を参照。

## 2. 収集手順

### ステップ1：情報源の一次スキャン
「3. 情報源の優先順位」に従い、優先度の高い情報源から順に確認する。

### ステップ2：候補案件のリストアップ
対象条件（PROJECT.mdの初期対象）に合致しそうな発表・報道を候補として仮リストにする。
この段階では厳密な判定は行わず、広めに拾う。

### ステップ3：対象・対象外の判定
候補案件それぞれについて、PROJECT.mdの「初期対象外」に該当しないかを確認する。
該当する場合は候補から除外し、除外理由を簡潔に記録しておく（cases.csvには載せない）。

### ステップ4：重複・続報の確認
新規案件として登録する前に、「5. 重複判定方法」に従って既存レコードと同一案件でないかを確認する。

### ステップ5：cases.csvへの登録
data/validation/README.md の列定義・入力ルールに従い、1案件につき1行を登録する。
不明な項目は unknown または未確認とし、推測で埋めない（CLAUDE.md の運用ルールに従う）。

### ステップ6：出典の確認
各案件について、可能な限り一次情報（自治体・企業の公式発表）を確認し、
source_url_1 / source_type_1（、確認できれば source_url_2 / source_type_2）に記録する。

### ステップ7：人間レビュー
すべてのレコードは human_reviewed が確認されるまで下書き扱いとする。
AIによる収集・入力のみでは確定情報として扱わない。

### ステップ8：50件到達後の評価
50件（またはそれ以上）を収集した時点で「6. 検証結果の評価方法」に従って合否を判定する。

## 3. 情報源の優先順位

信頼性・一次性の高い順に確認する。

1. **自治体の公式発表**
   都道府県・市区町村・一部事務組合の公式サイト、報道発表資料、議会資料、入札・調達公告
2. **企業の公式発表**
   生成AI関連企業・ベンダーのプレスリリース、導入事例ページ
3. **官公庁・関連団体の発表**
   デジタル庁、総務省、地方公共団体情報システム機構（J-LIS）等による事例集・調査
4. **信頼性の高い報道機関の記事**
   全国紙・地方紙・業界専門メディアによる報道（一次情報へのリンクがあるものを優先）
5. **その他の報道・ブログ・SNS**
   一次情報が確認できない場合の参考情報にとどめ、単独では登録の根拠としない
   （少なくとも1〜5のいずれかで自治体名・企業名が確認できることを登録の前提とする）

同一案件について複数の情報源が確認できる場合は、source_url_1 に最も一次性の高いものを、
source_url_2 に次点のものを記録する。

## 4. 判定基準

### 「自治体と企業を特定できる」
- municipality（またはprefecture）と companies の両方に具体的な名称が入力できる場合に該当する。
- 企業名が不明でも自治体名が確認できるだけでは該当しない。

### 「製品またはサービスを特定できる」
- product_name に具体的な製品名・サービス名（例：ChatGPT、Azure OpenAI Service、特定の自治体向けAIサービス名など）を入力できる場合に該当する。
- 「生成AI」「AIチャットボット」といった一般名詞のみの場合は該当しない。

### 「案件の進捗段階を判定できる」
- current_status に、指定のステータス一覧（下記）から「情報収集中」「続報なし」以外の
  具体的な段階を割り当てられる場合に該当する。
- 情報が乏しく「情報収集中」のままの案件は、この条件には数えない。

### 「複数の公式出典を確認できる」
- source_url_1 と source_url_2 の両方が入力されており、かつ両方が
  自治体公式・企業公式・官公庁のいずれか（情報源の優先順位1〜3）に該当する場合。

### 「同一案件の続報を確認できる」
- 案件の発表以降、時間を置いた別の発表・報道（ステータスの進展や結果の公表など）が
  確認でき、それが「5. 重複判定方法」に従って同一案件として1レコードにまとめられている場合。
- notes 欄に続報の概要と日付を記録する。

### 「費用または定量成果を確認できる」
- contract_amount_yen に具体的な金額が入力できる、または
  quantitative_result に具体的な数値（削減時間、利用件数、満足度スコアなど）が
  入力できる場合に該当する。

## 5. 重複判定方法

新しい候補案件を登録する前に、既存の cases.csv の中に同一案件がないかを以下の順で確認する。

1. **municipality（自治体名）と department（担当部署）が一致するか**
2. **companies（企業名）または product_name（製品名）が一致するか**
3. **use_case（用途）が実質的に同一の取り組みを指しているか**

上記1〜3がいずれも一致・類似し、かつ発表内容が「新しい局面の追加」（例：実証→本導入、
契約→効果検証の公表）であると判断できる場合は、新規案件として別の case_id を発行せず、
既存レコードの current_status・end_date・source_url_2・notes 等を更新し、
続報であることが分かるよう notes に日付と概要を追記する。

一方で、同一自治体・同一企業であっても、対象とする業務・部署・用途が明確に異なる場合
（例：総務部門でのAI議事録作成と、教育委員会でのAI教材開発は別の取り組み）は、
別案件として新規の case_id を発行する。

判断に迷う場合は、AIの判断のみで確定せず、notes に判断に迷った旨を記録し、
人間のレビューを仰ぐ（CLAUDE.md の運用ルールに従う）。

## 6. ステータス一覧

current_status には以下のいずれかを入力する。

- 情報収集中
- 検討・方針策定
- 連携協定
- 実証予定
- 実証中
- 実証結果公表
- 調達・公募
- 契約・採択
- 本導入
- 共同利用
- 効果検証
- 他自治体展開
- 終了
- 続報なし

## 7. 検証結果の評価方法

### 集計方法
50件（またはそれ以上）の収集が完了した時点で、cases.csv を基に
「1. 検証目標と合格条件」の6条件それぞれについて該当件数を集計する。

### 評価区分
- **合格**：6条件すべてを満たす場合。データベース事業として次段階（サイト化・拡張）の検討に進める。
- **条件付き合格**：4〜5条件を満たす場合。未達の条件を分析し、
  対象範囲の見直しや情報源の追加で改善余地があるかを検討する。
- **不合格**：3条件以下しか満たさない場合。現行の対象範囲・情報源では
  データベース事業として十分な密度・継続性が得られないと判断し、
  対象範囲の再定義または企画自体の見直しを行う。

### 評価時に記録する内容
- 条件ごとの達成件数と未達の理由（情報源が乏しい、自治体側が非公開にしている、等）
- 収集にかかった時間・作業量の目安（継続運用時の工数見積もりの参考とする）
- 対象範囲・対象外条件の妥当性についての所感（広すぎる／狭すぎる等）

評価結果は本ドキュメントまたは別途のレビュー記録にまとめ、
次の意思決定（PROJECT.md の「将来構想」に進むか、対象範囲を見直すか）の材料とする。
